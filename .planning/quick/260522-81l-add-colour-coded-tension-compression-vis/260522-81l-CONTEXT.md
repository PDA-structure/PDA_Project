<decisions>
## Implementation Decisions

### 1. Visible legend — placement: **Canvas corner overlay**

Small key pinned to the top-right of the canvas drawing area showing:
- Blue line → "Tension (+)"
- Red line → "Compression (−)"
- Grey line → "Near-zero"
- Purple arrow → "Reaction" (once arrows ship — same legend block)
- Thickness scale hint → optional "thick = high |F|"

**Why this placement:** lives with the drawing so it appears in any canvas-derived screenshot/export the user puts in a calc report. Doesn't compete for results-panel real estate.

**Open implementation question to resolve in the plan:** is the legend
(a) absolute-positioned HTML overlaid on the canvas (easier, NOT captured if someone right-click-saves the canvas image), or
(b) drawn on the canvas itself via a `drawLegend()` function (more work, IS captured in canvas exports).
**Recommended:** (b) — canvas-drawn — because the user's stated intent for colour-coding is calc-report use, and report screenshots come from the canvas image. The planner should pick (b) unless there's a strong reason against.

### 2. Line-thickness scaling — rule: **Linear, 1.5 px → 6 px**

```
thickness = 1.5 + 4.5 * (|F| / max|F|)
```

- `max|F|` computed across all members for the current solve (single normalisation pass after the solve, NOT per-frame).
- Members with `|F| < 1e-3` (the existing zero-tolerance) clamp to the minimum 1.5 px.
- Applied inside the existing `drawMembers()` per-member loop.
- Current uniform `lineWidth = 2` becomes the computed value.

**Why linear over sqrt/stepped:** linear is the standard FEM-viewer default, easiest to read at extremes, and pairs naturally with the existing intent of showing the load path. Sqrt and stepped were offered as alternatives for special cases — neither applies here.

### 3. Reaction arrows on canvas — colour: **Purple/magenta** (`#7b1fa2`)

- Drawn at every restrained DOF after a successful solve.
- Direction: arrow points in the direction the support exerts force on the structure (per `FG = Kp @ UG` convention: positive `FG[base+0]` = +X reaction, positive `FG[base+1]` = +Y reaction).
- Magnitude: shown as `<value> kN` label next to the arrow tail (mirrors the existing load-arrow label style).
- Length: scaled like loads (`24 * sc` base + magnitude-proportional, exact rule open for planner — keep it simple).
- Visibility: must respect `chkSupports` toggle? **No — reaction arrows are part of the solve output, not the input geometry. Tie visibility to `results !== null` only, NOT to the supports checkbox.** This keeps the geometry/output split clean.

**Why purple:** loads are green, members are blue/red, supports are dark navy `#1a1a2e`. Purple `#7b1fa2` is the only major hue not already used; reactions become instantly distinguishable. Black (option B) was less attention-grabbing. Same-as-loads dashed (option C) risked reading as "tentative".

### 4. Colour palette — **Keep current (`#1565c0` blue / `#b71c1c` red)**

User chose to keep the existing Material 800-level pair rather than refreshing to brighter Material 700 (`#1976d2` / `#d32f2f`) or switching to colour-blind-safe (`#0072B2` / `#D55E00`).

Implication: the four palette constants used in `drawMembers()` stay as-is. Only the *new* legend, thickness rule, and reaction-arrow logic land in this task. The CSS classes (`.tension`, `.compression`, `.zero-force`) should match the canvas colours so the tables and canvas agree visually — planner should confirm `ui/truss2d/style.css` already uses the same hex codes; if not, align them in the same atomic commit.

### Claude's Discretion

- Exact legend layout (vertical list vs horizontal pill) and font sizing — pick something that looks right at default symbol scale, scale with `getSymbolScale()` like other on-canvas elements.
- Exact reaction-arrow length rule (uniform vs proportional to |F|/maxReaction) — the planner can recommend, but keep simple.
- Reaction-arrow zero-tolerance (skip arrows for `|FG[idx]| < 1e-3` to avoid drawing zero-length nothings) — match the member zero-tolerance.
- Whether the legend includes a "thick = high |F|" hint or omits the thickness key (cleaner without it; user can intuit).
- Whether thickness scaling applies to the deflected-shape preview as well, or only to undeformed members (recommend: only undeformed — deflected is a separate dashed overlay).

</decisions>
