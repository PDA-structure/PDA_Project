# Quick Task 260523-i52: Add floating in-window panels to frame2d — Context

**Gathered:** 2026-05-23
**Status:** Ready for planning

<domain>
## Task Boundary

**Add a float/dock affordance to every card in the frame2d toolbar.** Each `<details class="card">` element gains a small button in its `<summary>` header. Clicking the button pops the card out of the toolbar into a `position: absolute` floating panel inside the same browser window, draggable by header. Clicking again returns it to its toolbar slot. State (checkbox values, slider positions, draw() wiring) preserved across both transitions.

**Out of scope (explicitly deferred to a future quick task):**
- Cross-window pop-out via `window.open()` (second-monitor support) — the more expensive Option 2 from the side-question discussion. Floating-in-window is the cheap-and-impactful path; if a real second-monitor need emerges after using floating panels, a separate quick task can layer pop-out on top of the float/dock infrastructure.
- Truss2d mirror — once frame2d's floating-panel pattern is validated through UAT, a separate quick task ports the same affordance to `ui/truss2d/` (same shape as how 260523-cyp mirrored 260522-81l).
- Resize handles on floated cards — keep float width identical to docked width for v1.
- Mobile/touch event support — desktop-first.
- Keyboard accessibility (focus + arrow-key drag) — desktop pointer-first.

</domain>

<decisions>
## Implementation Decisions

### D-1. Float affordance scope — **All 10 cards uniformly**

Every `<details class="card">` element in `ui/frame2d/index.html` gets a float button in its `<summary>`. Cards: Geometry, Supports, Node Loads, Member Loads, Member Properties, Edit, File, Material Properties, Section Calculator, Display.

**Why uniform:** consistent affordance = discoverable. The engineer doesn't need to remember "which cards can float" — every card can. Implementation cost is the same since the float logic is a single generic helper.

### D-2. Float origin — **Top-right of canvas area**

When clicked, the card pops to `position: absolute` at the top-right corner of the canvas region. The user can then drag from anywhere they want. Choosing a stable corner means engineer can predict where to look after clicking float; choosing the canvas area (not the browser corner) means the card stays "in the work zone" near what it controls.

**Specifics:** position relative to the `.canvas-area` bounding rect, with a small inner margin (e.g., 12 px) so the card doesn't kiss the edges.

### D-3. Re-dock — **Manual button only**

The same button that floated the card also docks it. When docked, the button shows `↗` ("float"); when floated, it shows `↙` ("dock"). Single click toggles. **No auto-snap on drag-over-toolbar** — engineer can safely park the floated card near the toolbar without accidentally re-docking it.

### D-4. Persistence — **None; reset to all-docked on reload**

No localStorage. Fresh page load = every card docked. Keeps the system stateless and avoids localStorage corruption / off-screen-after-resize / cross-tab edge cases. Engineer gets a clean default each session; if they want a specific layout repeatedly they can build muscle-memory for the float clicks.

### D-5. Float button glyph — **Unicode arrows**

Floated state shows `↙` (south-west arrow = "dock back to toolbar on the left"). Docked state shows `↗` (north-east arrow = "float out to the canvas"). `title=` attribute reads "Float panel" / "Dock to toolbar". Small button (≈18 px square), right-aligned within the `<summary>`, separated from the chevron disclosure arrow by `auto` margin.

**Why Unicode over SVG:** zero asset cost, theme-aware (uses surrounding `color` token via inherit), matches existing emoji-light frame2d aesthetic (themeToggle button uses `☀/☾` Unicode).

### D-6. Drag handle — **`<summary>` element only**

The card header `<summary>` is the drag handle. The card body (form controls, sliders, checkboxes) is NOT draggable — interacting with body elements should never accidentally drag the panel. The cursor on `<summary>` switches to `grab` / `grabbing` when the card is in the floated state. When docked, cursor stays as `pointer` (existing accordion behaviour).

### D-7. Z-index stacking — **Bring-to-top on drag start AND float click**

When multiple cards are floated, the most-recently-activated one sits on top. Activation = either clicking the float button OR starting a drag on the header. A small `_floatZIndex` counter increments each activation. No global stacking complexity beyond that.

### D-8. Drag containment — **Clamp to browser window**

While dragging, the card's top-left corner is clamped so the card cannot be dragged entirely off-screen. Minimum 40 px of the card stays visible on each edge. Prevents "I dragged it off and now I can't find it" support requests.

### D-9. Card width — **Identical to docked width**

Floated card uses the same width it had in the toolbar (frame2d's `.card { min-width: 110px }` per 260505-u2h). No resize handles. Keeps v1 simple. If width-tuning becomes a real ask, that's a separate iteration.

### D-10. Visual cue for floated state — **Drop shadow + slightly stronger border**

A floated card carries a subtle drop shadow + slightly thicker/darker border than its docked twin so it visually "lifts" off the canvas like the legend card does (260523-cyp T11 set the precedent). Token-driven via existing `--color-shadow` and `--color-border-input` so light/dark themes are covered automatically. **Specifics open to discretion** — planner picks shadow opacity/offset/blur to match the legend card aesthetic.

### Claude's Discretion

- Whether floated card gets its own DOM element (cloned from original) OR moves the original element in the DOM (`appendChild` to a new parent). **Recommend: move the original.** Listeners + state preserved automatically, no double-element-id sync. The toolbar slot is filled by a placeholder during float so docking restores the original position.
- How the placeholder behaves visually — empty 0-height slot vs collapsed-card stub vs nothing at all. **Recommend: no placeholder, just remove from toolbar flow.** Adjacent cards re-flow upward to fill the gap. Docking re-inserts at the card's original index (tracked via `data-original-index` attribute set at page load).
- Pointer-event behaviour during drag — `pointer-events: none` on canvas while dragging so the canvas doesn't capture mouse moves and break the drag. **Recommend yes**, restored on `mouseup`.
- Theme-aware drag-cursor — `grab` light mode, `grab` dark mode (same — Unicode cursor names are theme-neutral).

</decisions>

<specifics>
## Specific Ideas

### Files in scope
- `ui/frame2d/script.js` — new `setupCardFloat()` function called at page-init time over every `<details class="card">` element. Adds float button to summary, attaches mousedown→drag handlers, manages float/dock state machine, handles z-index bookkeeping.
- `ui/frame2d/style.css` — new `.card.floating` rule (position: absolute, z-index, drop shadow, border tweak), `.card-float-btn` rule (small button styling), placeholder behaviour, drag-cursor states.
- `ui/frame2d/index.html` — no edits expected. Float buttons are JS-injected so HTML stays clean. (If HTML edits ARE needed for some reason, only adding `data-original-index` attributes — no structural changes.)

### Files explicitly NOT in scope
- `ui/truss2d/*` — mirror task is separate.
- `solver_core/**`, `api_server/**`, `tests/**` — UI-only change.

### State machine
Each card has 2 states: `docked` (default) and `floating`. Single button toggles. State stored on the DOM element as `data-state="docked"` or `data-state="floating"`. Helpers `floatCard(card)` and `dockCard(card)` handle the transitions.

### Drag handlers
- `mousedown` on `<summary>` when card is `floating`: capture start position, register `mousemove` + `mouseup` listeners on `document`, set z-index to top.
- `mousemove`: update `transform: translate(x, y)` or absolute `left/top`. Clamp to window bounds per D-8.
- `mouseup`: remove document listeners, release.
- Native `<summary>` click-to-toggle-disclosure behaviour preserved by NOT calling `preventDefault()` unless the mouse has actually moved more than ~3 px (drag threshold).

### Float-button event behaviour
- `click` on the float button: toggle state. `stopPropagation()` so the click doesn't bubble to the `<summary>` and toggle the disclosure.
- Button is appended/recognised by class; native button so keyboard Enter activates.

</specifics>
