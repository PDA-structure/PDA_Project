<objective>
Light-mode contrast pass for frame2d UI — fix two readability issues identified during work-laptop UAT 2026-05-05 on a light-background secondary screen.

**Problem 1:** Light-mode canvas gridlines are nearly invisible (`--color-grid: #eeeeee` against `--color-canvas-bg: #fefefe` — only ~1% luminance difference).

**Problem 2:** UDL and Spring floating panels (`#udlPanel`, `#springPanel`) use inline `style="background:#fff; border:1px solid #90caf9; ..."` attributes with hard-coded colours, plus inline label/input/button styles. These bypass the design-token system entirely, so they don't follow the rest of the theme and look out of place / hard to read in light mode.

**Out of scope (explicit clarification from discovery):**

The planning context asked to apply "identical fix" to `ui/truss2d/style.css`, but discovery shows:
- **truss2d/style.css has NO design tokens** — it is the older pre-j8m stylesheet with hard-coded hex colours throughout (`#f0f2f5`, `#1a1a2e`, etc.). There is no `--color-grid` to retune.
- **truss2d/index.html has NO #udlPanel or #springPanel** — those features are frame-specific.
- **truss2d's canvas grid colour is in `script.js` line 361** (`ctx.strokeStyle = '#eee'`), not CSS — and the constraints explicitly say "No JS changes".

Therefore truss2d is left untouched in this task. Recommend capturing as follow-up backlog: "Migrate truss2d UI to frame2d's design-token system (j8m parity for truss2d)" — that is a much larger restyle, not a contrast pass.

Purpose: Make frame2d UI usable on a light-mode work laptop without regressing the existing 13/13-UAT-approved dark mode.
Output: `ui/frame2d/style.css` (token retune + new panel rules) and `ui/frame2d/index.html` (inline-style cleanup on #udlPanel + #springPanel).
</objective>
