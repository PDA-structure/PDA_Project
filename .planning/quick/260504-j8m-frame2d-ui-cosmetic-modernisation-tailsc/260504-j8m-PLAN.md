### View transform (infinite grid prerequisite)
```js
let view = { scale: 1, tx: 0, ty: 0 };

// canvas px → world coord
function toWorld(clientX, clientY) {
  const rect = canvas.getBoundingClientRect();
  const px = (clientX - rect.left) * (canvas.width / rect.width);
  const py = (clientY - rect.top)  * (canvas.height / rect.height);
  return { x: (px - view.tx) / view.scale, y: (py - view.ty) / view.scale };
}
```

In `draw()` (line 740-742) the executor sets:
```js
ctx.setTransform(1, 0, 0, 1, 0, 0);
ctx.clearRect(0, 0, canvas.width, canvas.height);
ctx.setTransform(view.scale, 0, 0, view.scale, view.tx, view.ty);
drawGrid();
```

So drawGrid runs with the world transform active. Loop bounds therefore must be in WORLD space:
```js
const worldLeft   = (0              - view.tx) / view.scale;
const worldRight  = (canvas.width   - view.tx) / view.scale;
const worldTop    = (0              - view.ty) / view.scale;
const worldBottom = (canvas.height  - view.ty) / view.scale;
```

### Existing drawGrid (script.js 767-776) — what to replace
```js
function drawGrid() {
  ctx.strokeStyle = '#eee';
  ctx.lineWidth = 0.5;
  for (let x = 0; x < canvas.width; x += GRID) {
    ctx.beginPath(); ctx.moveTo(x, 0); ctx.lineTo(x, canvas.height); ctx.stroke();
  }
  for (let y = 0; y < canvas.height; y += GRID) {
    ctx.beginPath(); ctx.moveTo(0, y); ctx.lineTo(canvas.width, y); ctx.stroke();
  }
}
```
The bug-by-design here is that loop bounds are in CANVAS-pixel space, but the active transform is the world transform — so once the user pans far enough, the gridlines walk off the visible area.

**Goal:** replace the canvas-pixel-bounded grid loop in `drawGrid()` (lines 767-776) with a world-space loop that inverse-transforms canvas corners through the active `view` transform. **Step 1 — Replace `drawGrid()` body.**

Current implementation (script.js 767-776):
```js
function drawGrid() {
  ctx.strokeStyle = '#eee';
  ctx.lineWidth = 0.5;
  for (let x = 0; x < canvas.width; x += GRID) {
    ctx.beginPath(); ctx.moveTo(x, 0); ctx.lineTo(x, canvas.height); ctx.stroke();
```

Replace with:
```js
function drawGrid() {
  // drawGrid runs with the world transform active (set in draw()).
  // Inverse-transform canvas corners → world rect → snap to GRID multiples.
  ctx.strokeStyle = '#eee';
  ctx.lineWidth = 0.5 / view.scale;   // keep lines visually 0.5 px regardless of zoom

  const worldLeft   = (0            - view.tx) / view.scale;
  const worldRight  = (canvas.width - view.tx) / view.scale;
  const worldTop    = (0            - view.ty) / view.scale;
  const worldBottom = (canvas.height- view.ty) / view.scale;

  const xStart = Math.floor(worldLeft  / GRID) * GRID;
  const xEnd   = Math.ceil (worldRight / GRID) * GRID;
  const yStart = Math.floor(worldTop   / GRID) * GRID;
  const yEnd   = Math.ceil (worldBottom/ GRID) * GRID;

  for (let x = xStart; x <= xEnd; x += GRID) {
    ctx.beginPath();
    ctx.moveTo(x, worldTop);
    ctx.lineTo(x, worldBottom);
    ctx.stroke();
  }
  for (let y = yStart; y <= yEnd; y += GRID) {
    ctx.beginPath();
    ctx.moveTo(worldLeft, y);
    ctx.lineTo(worldRight, y);
    ctx.stroke();
  }
}
```

**Why this is correct:**
- `draw()` sets `ctx.setTransform(view.scale, 0, 0, view.scale, view.tx, view.ty)` BEFORE calling `drawGrid()` (line 742). Therefore drawGrid receives world-coord arguments through the canvas API.
- World corners come from inverse-transforming `[0, 0] → [canvas.width, canvas.height]` (canvas-pixel space) through the same transform, which is `world = (canvasPx - tx) / scale` per axis.
- Snapping each world bound to the nearest GRID multiple via `Math.floor` / `Math.ceil` ensures the grid lines align to the world origin, exactly as the original implementation did at zoom=1, pan=0.
- The `lineWidth = 0.5 / view.scale` keeps gridlines visually thin at any zoom — at the current zoom-of-1 default this is 0.5, identical to before. (If the executor judges this is a behaviour change beyond cosmetic, the line-width adjustment can be dropped — but in practice without it the grid becomes a solid block at high zoom-out, which is worse than the current behaviour.)

**Edge case sanity:**
- Worst-case panning to `view.tx = -100000` makes `worldLeft = 100000` and `worldRight = 100000 + canvas.width / view.scale`. Loop step `GRID = 20`. Number of vertical lines drawn ≈ `(canvas.width / view.scale) / GRID + 1`. At default scale = 1, canvas.width = 1000, that's 51 lines — same as today. At scale 0.1 (zoomed way out), 501 lines — still trivial. No performance concern.
- At scale 100 (extreme zoom in), only ~1-2 lines visible. Fine.
