## Pre-discussion scoping (before /gsd-discuss-phase)

Extended scoping conversation grounded in research of Autodesk Robot Structural Analysis and
Tekla Structural Designer. Key findings: the **load case is the atomic solvable unit**;
**combinations are an optional factored post-process**; the quick-analysis path is preserved
structurally by an **implicit default case**. Linear superposition makes combinations a
zero-re-solve layer.

Locked before formal discussion:
- Load case = nature-tagged container; natures = code-agnostic physical types (Self weight,
  Dead, Imposed, Wind now); invisible default active case.
- Manual table + EN 1990 6.10 auto-gen (leading/accompanying ψ₀), enveloped.
- Per-member governing-combination traceability (member→combo + reverse index) = the
  differentiator vs Tekla.
- Code-pluggable code packs (Eurocode first, BS next, then ASCE/NBR); code+country once.
- Multi-code natures stay physical so switching code never re-tags loads.
- truss2d first; portable to frame2d + 3D.

## User refinement: load cases

**User's choice:** Natures NOW = Self weight, Dead, Imposed, Wind. "Services" moved to the

## User refinement: the differentiator

**User's note:** TSD's failure is that with a large combination list there's no way to
identify which combination is critical for a given member. Capture this NOW as a first-class
feature — it becomes the most valuable asset when scaling to 3D complex models. Saved to
memory `loadcomb-critical-combination-traceability`.

## User refinement: multi-code + generator UX
