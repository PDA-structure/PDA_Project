# Phase 999.2: Load Combination Generator - Context

**Gathered:** 2026-06-27
**Status:** Ready for planning

<domain>
## Phase Boundary

Deliver **typed load cases + a code-pluggable load-combination generator** on the
truss2d tool, built so the model, engine, generator UX, and export format transfer
cleanly to frame2d and the future 3D solvers.

Scope of THIS phase:
- Load cases as nature-tagged containers (natures: Self weight, Dead, Imposed, Wind).
- An invisible **default active case** so the existing quick-solve path is unchanged.
- A **solver-agnostic combination engine in `solver_core`** (code-pack registry +
  generator + superposition + envelope + per-member governing-combination traceability),
  exposed via an additive API endpoint.
- EN 1990 **6.10** auto-generation with leading/accompanying ψ₀, for **STR-ULS** and
  **SLS-characteristic (γ=1.0)**.
- A manual combination table + a TSD-style two-page generator wizard with `(i)` info notes.
- Per-member **governing-combination traceability** surfaced in results + carried into the
  EC3 export contract.

OUT of scope (designed-for, generated/built later): wind uplift / favourable-permanent
(γG,inf=1.0) + EQU family; SLS frequent/quasi-permanent; GEO, Accidental/Seismic families;
6.10a/6.10b; BS/ASCE/NBR code packs (the pluggable seam is built now, only the Eurocode
pack is implemented); the later natures (Services, Roof Imposed, Snow, Snow Drift, Seismic,
Temperature, Settlement); self-weight auto-computation (nature exists as a placeholder, like
the frame2d self-weight feature to come).
</domain>

<decisions>
## Implementation Decisions

### Load case model
- **D-01:** A load case is a **named, nature-tagged container** of loads. Loads carry a
  `caseId`; factors NEVER live on loads or cases — only on combinations.
- **D-02:** Natures are **code-agnostic PHYSICAL types**. NOW: Self weight, Dead, Imposed,
  Wind. The enum is designed to grow (later: Services, Imposed-other, Roof Imposed, Snow,
  Snow Drift, Seismic, Temperature, Settlement). A load is tagged with a physical nature
  ONCE; the active code pack interprets it — switching code/country re-generates
  combinations WITHOUT re-tagging any load.
- **D-03:** The model always has a **default active case** (default nature = Dead, since
  Self weight is auto-computed and not where loads are hand-placed). A user can drop a load
  and solve without ever opening the case table → **quick-solve path is never bottlenecked.**
- **D-04:** Self weight is a defined nature/placeholder now; **auto-computation is deferred**
  (mirrors the frame2d self-weight feature). In 2D there are no slabs; the "self-weight
  excludes slabs" rule matters in the future 3D solvers.

### Combination engine — location & architecture
- **D-05:** The combination engine lives in **`solver_core` as a solver-agnostic shared
  layer**: a code-pack registry (natures→categories, γ, ψ₀/ψ₁/ψ₂, generation rules), a
  generator that emits combination definitions, and the superposition + envelope +
  governing-combo math. Pure numpy/python, **no matplotlib, no printing** (project hard rule).
- **D-06:** Exposed via an **additive API endpoint** (e.g. `POST /solve/<solver>/combinations`
  or a combinations endpoint that orchestrates per-case solves). The **existing single-case
  `/solve/truss2d` is untouched** — quick path preserved.
- **D-07:** The engine is **solver-agnostic** — it needs only per-case member forces, so
  truss2d, frame2d, and future 3D plug in unchanged. The marimo calc platform can `import`
  the package directly (single source of truth for γ/ψ — no JS↔Python drift).
- **D-08:** The **UI owns only** the case table + generator wizard and CALLS the engine. It
  never re-implements factor logic.
- **D-09:** Linear superposition: solve each non-empty case once, cache, combine by factors
  — **no re-solve** when forming/editing combinations.

### Code-pluggability
- **D-10:** A **code pack** abstraction owns everything code-specific: nature→category
  mapping, partial factors γ, combination factors ψ, and generation rules (Eurocode iterates
  leading variable w/ ψ₀; BS/ASCE use fixed enumerated combos — different logic, same output
  shape). National Annex = a variant overlay overriding NDP factor values.
- **D-11:** Build the **code-pack seam now; implement the Eurocode pack only.** BS 5950 /
  BS 6399 next, then ASCE 7 / NBR (Brazil). Code + country/NA chosen once (Robot-style).

### Auto-generation & combination families
- **D-12:** EN 1990 **6.10** for **STR-ULS** AND **SLS-characteristic (γ=1.0)** are
  generated in this phase. 6.10 generates multiple combinations by iterating each variable
  action as leading (full γQ) with others reduced by ψ₀ (accompanying); degrades gracefully
  to a single `1.35G + 1.5Q` row when only Dead + Imposed exist.
- **D-13:** SLS-characteristic uses all partial factors = 1.0 (1.0·ΣG + 1.0·Q_lead +
  ψ₀·Q_accomp) — it drives the truss δ_max deflection check already displayed.

### Generator UX (TSD-style)
- **D-14:** Two-page wizard. **Page 1 (Parameters):** `Code ▾` + `Country/NA ▾` (once) +
  family/limit-state checkboxes + equation set (`6.10` now, `6.10a/b` later) + a **`Next ▸`**
  button. **Every option has an `(i)` info popup** citing the clause + when to use.
- **D-15:** Families shown from day one (TSD parity, never restructured later): **STR-ULS
  (☑ default), SLS-characteristic (☑ default)**; SLS frequent/quasi-permanent, EQU, GEO,
  Accidental/Seismic shown **visible-but-deferred** (off / greyed "coming soon").
- **D-16:** **Page 2 (Preview):** generated rows grouped by family/class, each **editable**
  (toggle on/off, rename, inline-edit any factor), shown with the human-readable factored
  expression + a plain leading-action tag ("Imposed leading"). `[◂ Back] [Generate → table]`.
- **D-17:** Generated rows drop into the **same editable combination table** as manual
  combinations (auto-gen = head-start, not a cage). Both paths share one data structure
  `combination = {name, class (STR/SLS/…), terms:[{caseId, factor}]}`.

### Governing-combination traceability (THE differentiator)
- **D-18:** Enveloping is **provenance-carrying**, not bare min/max. For each member store
  the worst force AND **which combination produced it** (governing for max tension; max
  compression once wind-uplift lands). This is the Tekla pain being designed out and the key
  asset for 3D.
- **D-19:** **Bidirectional visibility:** member → governing combination + full per-combo
  breakdown; AND a reverse index combination → members it governs.
- **D-20:** Results UI gains a selector across **[per-case] / [per-combination] / [ENVELOPE]**;
  the member results row shows force | sense | **governing combination**.
- **D-21:** **Export contract:** the SEED-005 handoff `load_combination` field becomes the
  **per-member governing-combination string** (e.g. "ULS-2: 1.35G + 1.5W + 1.05·Imposed"),
  feeding the EC3 tension sheet. Extends the single-string field already wired — no breaking
  change to the marimo reader.

### Portability
- **D-22:** Build + test on **truss2d first**, but the load-case model, engine/code-pack,
  generator UX, and export format MUST transfer cleanly to **frame2d** (same format) and the
  3D solvers. Prefer shared/portable structures over truss-only ones.

### Claude's Discretion
- Exact module path/name within `solver_core` (e.g. `loads/combinations.py` vs a `combinations/`
  package) and the precise endpoint shape — planner/researcher to choose, honouring D-05..D-08.
- Save/Load interchange format extension for cases + combinations; **legacy untyped saved
  models default all loads to the Dead case** on load (backward-compatible).
- Canvas display of per-case loads (e.g. colour-by-nature, show-active-only toggle) — choose
  a clean approach consistent with existing truss2d panel/label conventions.
- Placement of the case table within the existing truss2d floating panel-section layout.
</decisions>

- **Linear solver + adapter pipeline** (truss2d.py → Truss2DAdapter → AnalysisResult): the
  superposition basis — solve each case through the existing path, combine forces after.
- **AnalysisEngine registry** (engine.register): combination endpoint reuses dispatch.
- **truss2d export** (`exportAnalysis()` already emits `tension_members`/`compression_members`,
  per-member A, sense, and a `load_combination` string) — extend the string to per-member
  governing combo; the SEED-005 marimo reader already binds to this shape.

  the case table + generator here.

  per-member override pattern — the model for per-case structures done cleanly.

- Analytical-verification test pattern (pytest) — combination engine needs analytical cases
  (e.g. 1.35G+1.5Q hand-check; leading/accompanying ψ₀ expansion; SLS γ=1.0).

<specifics>
## Specific Ideas

- Generator modelled on **Tekla Structural Designer's combination generator** (family-based,
  STR/SLS/EQU/GEO/Accidental/Seismic), but designing OUT TSD's key failure: **inability to see
  which combination governs each member.** Per-member governing-combo + reverse index is the
  explicit differentiator and the most-valuable-asset target for 3D.
- `(i)` info popups on every generator option (clause + when-to-use) → self-documenting UI.
- Robot Structural Analysis pattern confirmed: load case = atomic solvable unit; combination =
  optional factored post-process; code + NA chosen once in job preferences.
</specifics>
