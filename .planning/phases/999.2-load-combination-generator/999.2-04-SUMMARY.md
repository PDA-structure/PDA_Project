# Phase 999.2 Plan 04: truss2d Combination Table + Generator Wizard Summary

**A shared editable combination table (manual + generated, one data shape) plus a two-page TSD-style generator wizard for the truss2d UI: the wizard calls the Wave-2 engine generate-only (no gamma/psi in JS, D-08), generated rows drop into the same table as manual rows (D-17), and editing a factor or toggling a row re-superposes + re-envelopes cached per-case forces in pure JS with no re-solve and no network (D-09) — with reusable (i) popovers citing EN 1990 clauses (D-14) and the deferred families/equation sets rendered greyed "coming soon" (D-15).**

## Accomplishments

### Task 1 — Combination table (C3) + JS superpose/envelope
- **Shared model** (`script.js`): `combinations = []` (rows `{id,name,cls,terms:[{caseId,factor}],on,generated,leading}`) + `perCaseForces = {}` cache (caseId → member_forces) + `comboEnvelope` carried for Wave 5.
- **Pure-JS helpers**: `superposeJS(combo)` (Σ factor·perCaseForces[caseId], zero vector for missing cases), `envelopeJS(activeCombos)` (per-member argmax/argmin governing combo + reverse-index `governs` count), `recombine()` (re-superpose all `on` rows + refresh badges IN PLACE — **no fetch/XHR**).
- **Combinations panel** (`index.html`): new floatable `.panel-section` "Combinations" below Load Cases — table with on/off | name | class chip | terms (font-mono, inline-editable) | governs badge | delete ✕; header actions "Generate combinations…", "Add row", "Clear generated" (danger). Empty-state copy per UI-SPEC.
- **Edit handlers**: inline factor edit validates the number (T-9992-11 — blank/NaN shows "Enter a number for every factor." and is not committed) then calls `recombine()`; toggle re-renders; delete + clear-generated honour the UI-SPEC destructive copy; "Add row" seeds one term per case at factor 0.
- **CSS** (`style.css`): class chip (STR accent / SLS danger), font-mono term cells + factor inputs, governs badge, combo delete, empty state — all on existing tokens, no new spacing tokens.
- **Save/load**: `combinations` persisted under `canvas{}` (additive); restored default-on-read; `perCaseForces` never saved (recomputed from the engine).

### Task 2 — Generator wizard (C4) + (i) info-popover (C5)
- **Two-page wizard** (`index.html`): `.wizard-overlay` scrim + `.wizard-card` (~420px, reuses the floating-card blur/shadow family). Page 1 = Code select (Eurocode enabled; BS/ASCE/NBR disabled "coming soon"), Country/NA (UK enabled), family checkboxes (STR + SLS checked; SLS frequent, SLS quasi-permanent, EQU, GEO, Accidental/Seismic disabled greyed "coming soon"), equation set (6.10 on; 6.10a/b disabled). Page 2 = editable preview grouped by class with toggle/rename/inline-factor + leading-action tags + static degenerate/no-variable notes.
- **Controller** (`script.js`): `openWizard`/`closeWizard` (Esc + outside-click + ✕ dismiss; scrim adds `body.card-dragging` to block canvas); `wizardNext()` posts geometry+cases+`generate:{code:'eurocode_uk',families:[checked]}` to `/solve/truss2d/combinations`, caches `per_case`, builds editable preview rows with **terms reconstructed from the engine's `expression`** (no gamma/psi in JS — D-08); `wizardGenerateToTable()` pushes the (edited) rows into the shared `combinations` model flagged `generated:true`, then `renderCombinationTable()` + `recombine()`.
- **Reusable (i) popover**: one `#infoPopover` element; every `.info-btn` carries `data-info-title`/`data-info-body` (EN 1990 clause + when-to-use); one open at a time; `aria-expanded` toggled; Esc / outside-click dismiss.

- **CSS**: `.wizard-overlay`/`.wizard-card`/`.wizard-*`, `.info-btn` (16px accent circle, 24px-ish hit area), `.info-popover` (max-width 260px blur card); wizard primary buttons use `--color-accent-bright` (the SOLVE yellow stays the only yellow). Dark-theme values for the new blur cards.
- **A11y**: aria-labels on every glyph-only button (✕ close, delete ✕, (i)) and on the on/off + factor inputs.

## Decisions Made
- **Terms reconstructed client-side** from the engine's `expression` string (the endpoint returns `expression` but not `terms`, and this plan is UI-scoped). Factors are parsed straight from the engine output — the UI never computes gamma/psi (D-08). Nature symbols map back to cases consumed in case order; distinct-nature setups (Dead/Imposed/Wind — the real scenario and the UAT case) are unambiguous. See Known Limitations.
- **In-place badge refresh** in `recombine()` so a factor edit keeps input focus and never triggers a spinner or a full re-render.
- **Static info-note copy** in index.html keeps the "1.35·G + 1.5·Q" wording out of script.js so the wizard generate path carries zero factor literals (T-9992-12).

## Deviations from Plan
- **Tasks 1 and 2 committed together (`9c4a18a`).** The wizard (Task 2) reuses the shared `combinations`/`perCaseForces` model and the `buildTermsCell` / `fetchCombinations` / `recombine` helpers introduced in Task 1, and both tasks edit the same three files in interleaved regions. Whole-file git staging cannot cleanly separate them post-hoc, so they were delivered as one cohesive atomic commit rather than fabricating an artificial split. Both tasks' verify + acceptance criteria pass independently (see below).
- **Rule 2 (completeness):** added additive save/load persistence of `combinations` (CONTEXT discretion / PATTERNS (h)) and a one-shot per-case fetch on manual "Add row" when the cache is empty, so a manual-only workflow can also envelope.

## Known Limitations
- **Repeated-nature term mapping:** when two load cases share the same nature (e.g. two Wind cases), `reconstructTerms` assigns parsed expression entries to those cases in case order, which may not distinguish leading vs accompanying perfectly. Distinct-nature setups (the standard and the UAT scenario) are exact. The clean fix (the endpoint returning `terms`) is deferred to a future api_server change — out of this plan's UI-only scope.
