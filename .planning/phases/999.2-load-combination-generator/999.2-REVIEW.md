## Warnings

### WR-01: Per-case force vector overwrites loads on the same node + direction

**File:** `api_server/app.py:383-386`
**Issue:** The per-case force vector is built with assignment, not accumulation:

```python
fv = [0.0] * (2 * n_nodes)
for ld in case.loads:
    idx = ld.nodeId * 2 + (1 if ld.direction.lower() == "y" else 0)
    fv[idx] = ld.magnitude          # overwrites, not +=
```

If a single load case carries two loads on the same node and same direction (the
UI permits multiple point-load entries per node), the second silently clobbers the
first instead of summing. This is masked by the test oracle, which uses `+=`
(`tests/test_load_combinations.py:56`, `fv[idx] += ld["magnitude"]`), so the
divergence is never exercised. Note this mirrors a pre-existing pattern in the
single-solve path (`ui/truss2d/script.js:901`), so it is a latent
codebase convention rather than a new regression â€” but this is new code and the
correct accumulating form should be used here.

**Fix:**
```python
fv[idx] += ld.magnitude
```

### WR-02: Exported `force_kN` does not correspond to its `load_combination` label

**File:** `ui/truss2d/script.js:1861-1884`
**Issue:** In `exportAnalysis` (schema 1.2), each member row's `force_kN`/`force_N`
is taken from `results.member_forces` â€” the result of the **single** `/solve/truss2d`
solve, which sums every load across every case at factor 1.0 (unfactored,
characteristic). The per-member `load_combination` string, however, is the
**envelope governing combination** name from `governingComboFor(...)`. The two come
from different computations, so the reported force is not the force produced by the
named governing combination. The `sense` (`f > 0 ? 'T' : 'C'`) is likewise derived
from the single-solve sign, and is then used to pick `govT`/`govC` â€” so if the
true governing extreme is the opposite sense (e.g. a future wind-uplift case), the
label and the force disagree further. A downstream consumer (or engineer sizing a
member) may reasonably read `force_kN` as the design force for the named governing
combination, which it is not.

This appears to be a deliberate back-compat tradeoff (keep `force_kN` stable so the
SEED-005 marimo reader is undisturbed, add `load_combination` as additive
metadata), but the pairing is semantically misleading and should be either
documented in the export or switched to the governing combination's superposed
force.

**Fix:** When an envelope exists, populate `force_N`/`force_kN` from the governing
combination's superposed value (available via `superposeJS(govCombo)[idx]`) so the
force matches its label; or, if back-compat must be preserved, add an explicit
`governing_force_kN` field and document that `force_kN` is the characteristic
single-solve value.

## Info

### IN-01: Wizard reconstructs terms from the expression string â€” engine already has caseId

**File:** `ui/truss2d/script.js:2786-2804` (`reconstructTerms`), `api_server/app.py` (combinations response)
**Issue:** The `/combinations` response returns each generated combination as
`{name, cls, leading, expression, member_forces, delta_max_m}` but **omits the
terms**. The wizard therefore parses the human-readable `expression`
(`"1.35Â·Dead + 1.5Â·Wind"`) and maps each nature symbol back to a case, consuming
same-nature cases in order â€” the known, already-tracked limitation. Verified: for
the distinct-nature common case (one Dead, one Imposed, one Wind) each symbol
matches exactly one case (`matches.length === 1`), so results are correct; the
ambiguity only bites when two cases share a nature.

The engine `Combination` objects already carry `terms` (`case_id`, `factor`), and
the save/load path already round-trips caseId-based terms
(`ui/truss2d/script.js:2114-2116`). The easy, robust fix is to include them in the
response so the wizard never parses strings:

```python
"terms": [{"caseId": t.case_id, "factor": t.factor} for t in c.terms],
```

then `wizardNext` uses `c.terms` directly instead of `reconstructTerms(c.expression)`.
This eliminates the same-nature limitation at its root.

### IN-02: STR generator emits an empty-term ULS-1 when there are no permanent and no variable cases

**File:** `solver_core/src/pda_analysis_software/loads/combinations.py:117-122`
**Issue:** The `if not variable:` STR branch unconditionally appends a `ULS-1`
combination even when `permanent` is also empty, producing a combination with an
empty `terms` list. The SLS branch guards this case with `if terms:` (line 138).
The asymmetry is unreachable via the API (the `no_loaded_cases` 422 guard fires
before generation), but a direct `generate_eurocode` caller would get a spurious
empty combination.

**Fix:** Mirror the SLS guard â€” wrap the STR `not variable` append in `if terms:`.

### IN-03: `no_loaded_cases` guard relies on dict-truthiness; a None-force case would slip through

**File:** `api_server/app.py:403`
**Issue:** `if not per_case_forces:` treats the dict as empty-or-not, but a case
that solved with `member_forces is None` still occupies a key with a `None` value,
so the guard would not fire. `superpose` tolerates `None` (it skips via `.get`),
so no crash results, but the "n_cases_solved" count would include a case that
contributes nothing. In practice the truss solver always returns member forces, so
this is an edge case only.

**Fix (optional):** Count non-None entries, e.g.
`if not any(v is not None for v in per_case_forces.values()):`.
