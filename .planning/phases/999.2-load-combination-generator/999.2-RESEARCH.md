## Summary

This phase adds a **typed load-case layer** and a **solver-agnostic combination engine** on top of the existing, already-linear truss2d pipeline. The good news from reading the code: almost everything needed is already present in a form that makes superposition trivial. The truss solver is fully linear (`np.linalg.solve` on a fixed `Ks`), the adapter→engine→`AnalysisResult` pipeline is clean, and the UI already has the panel/float infrastructure (`setupCardFloat`), per-member-override precedent (`A_override`), and an export contract (`exportAnalysis`, schema 1.1) with a single `load_combination` string ready to be upgraded to per-member.

The core technical insight (D-09) is **linear superposition**: solve each non-empty load case once through the *existing* adapter (one `forceVector` per case), cache the resulting member-force vector, then form any combination as a weighted sum `N_combo = Σ factor_k · N_case_k` — pure arithmetic, no re-solve. This is mathematically exact for linear elastic FEM and is the reason the engine only needs "per-case member forces" to be solver-agnostic (D-07). Enveloping then reduces across combinations per member, carrying the governing combination's identity (D-18/D-19).

The EN 1990 factor values are well-established and citable (Table A1.1 for ψ, UK NA Table NA.A1.2(B) for γ). The one place needing user confirmation is **which imposed-load category** the default Dead+Imposed uses (drives ψ₀): the common assumption ψ₀=0.7 covers categories A–D and H, but storage (E) is 1.0. Recommend ψ₀=0.7 as the implemented default with the category exposed as a per-case property later.

**Primary recommendation:** Create a `solver_core/.../loads/` package (`code_packs.py` data-driven registry + `combinations.py` generator/superposition/envelope). Add one additive endpoint `POST /solve/truss2d/combinations` that orchestrates N per-case solves through the *unchanged* `Truss2DAdapter` and returns per-case forces + combination results + per-member envelope-with-provenance. Hardcode the Eurocode + UK NA pack from the cited tables, and cover it with analytical pytest cases (1.35G+1.5Q hand-check, ψ₀ leading/accompanying expansion, SLS γ=1.0, superposition-equals-direct linearity proof, envelope-picks-correct-combo).

### Pattern 1: Data-driven code pack (D-10/D-11)

A code pack is a **data object + a generation strategy**. Keep the *generic generator* free of code-specific logic; the pack carries factor tables AND the per-family generation rule (Eurocode iterates leading variable with ψ₀; BS/ASCE enumerate fixed combos — "different logic, same output shape", D-10).

```python
# Source: pattern grounded in CONTEXT D-10/D-11; numbers from EN 1990 / UK NA (see Factor Values)
from dataclasses import dataclass, field
from enum import Enum
from typing import Callable

class Nature(str, Enum):          # D-02 physical natures, str-valued for clean JSON
    SELF_WEIGHT = "Self weight"
    DEAD = "Dead"
    IMPOSED = "Imposed"
    WIND = "Wind"
    # growth: SNOW, ROOF_IMPOSED, SERVICES, SEISMIC, TEMPERATURE, SETTLEMENT ...

class ActionType(str, Enum):      # how the code treats a nature
    PERMANENT = "permanent"
    VARIABLE = "variable"

@dataclass(frozen=True)
class CodePack:
    name: str                                  # "Eurocode (EN 1990) — UK NA"
    nature_action: dict                        # Nature -> ActionType (D-10 nature->category)
    gamma: dict                                # {"G_sup":1.35,"G_inf":1.0,"Q_sup":1.5,"Q_inf":0.0}
    psi0: dict                                 # Nature -> ψ0 (leading/accompanying)
    psi1: dict = field(default_factory=dict)   # forward-compat (SLS frequent) — carried, not used
    psi2: dict = field(default_factory=dict)   # forward-compat (SLS quasi-permanent)

```

The Eurocode generator implements 6.10 iteration; a future `BS5950Pack` supplies a different `generate` but emits the **same `Combination` shape** (D-17), so the table/UI/envelope code never changes.

### Pattern 2: Linear superposition (D-09) — the load-bearing pattern

```python
# Source: standard linear-elastic FEM superposition; grounded in truss2d.py linearity
# Step 1 — solve each non-empty case ONCE through the existing pipeline (no solver change):
per_case_forces = {}          # caseId -> member_forces ndarray (N, tension +)
for case in cases:
    if not case.loads:
        continue
    fv = build_force_vector(case.loads, n_nodes)      # same shape as /solve/truss2d forceVector
    model = TrussModel2D(nodes, members, E, A, fv, restrainedDoF)
    result = engine.solve(model, "truss2d")           # UNCHANGED adapter/engine
    per_case_forces[case.id] = result.member_forces    # cache

# Step 2 — combine by factors (pure arithmetic, NO re-solve):
def superpose(combination, per_case_forces, n_members):
    N = np.zeros(n_members)
    for term in combination.terms:                    # term = {caseId, factor}
        N += term.factor * per_case_forces[term.caseId]
    return N
```

**Why exact:** for a fixed stiffness `Ks` and `Kp`, displacements are linear in `forceVector` (`u = Ks⁻¹ f`), and member force `N` is linear in `u` (truss2d.py:114). Therefore `N(Σ f_k) = Σ N(f_k)` and scaling commutes. This linearity is what makes the engine solver-agnostic (D-07): it only ever touches the per-case `member_forces` arrays.

### Pattern 3: Provenance-carrying envelope (D-18/D-19)

```python
# Source: CONTEXT D-18/D-19
# For each member: worst tension and worst compression, each tagged with the governing combo.
combo_forces = {c.name: superpose(c, per_case_forces, n_members) for c in combinations}
F = np.vstack([combo_forces[c.name] for c in combinations])   # shape (n_combos, n_members)

max_idx = np.argmax(F, axis=0)        # governing combo index for max tension per member
min_idx = np.argmin(F, axis=0)        # governing combo index for max compression per member
names = [c.name for c in combinations]

member_to_combo = [                    # D-19 forward map
    {"member": m+1,
     "max_tension_kN": F[max_idx[m], m]/1e3,  "gov_tension": names[max_idx[m]],
     "max_compression_kN": F[min_idx[m], m]/1e3, "gov_compression": names[min_idx[m]]}
    for m in range(n_members)
]
combo_to_members = {                    # D-19 reverse index
    name: [m+1 for m in range(n_members)
           if names[max_idx[m]] == name or names[min_idx[m]] == name]
    for name in names
}
```

In this phase (no wind uplift yet) the governing string written to the export is the **max-tension** governing combo (D-21 example "ULS-2: 1.35G + 1.5W + 1.05·Imposed"); the max-compression slot is wired now for the deferred uplift stage.

### Anti-Patterns to Avoid

- **Re-solving per combination** — violates D-09 and is wasteful; combinations are post-process arithmetic.
- **Putting factors on loads or cases** — D-01: factors live ONLY on combinations. A load case is unfactored characteristic actions.
- **Re-tagging loads when the code changes** — D-02: nature is physical and tagged once; the pack interprets it. Switching pack must NOT touch load tags.
- **Re-implementing factor logic in JS** — D-08: the UI calls the engine. The only JS that knows γ/ψ is what the engine returns (preview rows). Avoids JS↔Python drift (D-07).
- **Mutating `/solve/truss2d` or `Truss2DRequest`** — D-06 + no-regression gate. The combinations endpoint is strictly additive.
- **Printing or matplotlib in `loads/`** — CLAUDE.md hard rule.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Solving each case | A second solver | Existing `Truss2DAdapter` via `engine.solve(model,"truss2d")` | D-07; the adapter already returns `member_forces`; reuse keeps one FEM path |
| Per-case force vector | New assembly logic | The same flat-`forceVector` build already in `solve()` (script.js:545-550) and `app.py` | Identical shape; only the load *subset* per case differs |
| Envelope min/max + index | Manual loops | `np.argmax`/`np.argmin`/`np.vstack` | Vectorised, correct, and carries the index for provenance for free |
| Per-member override plumbing | New pattern | Mirror the `A_override` / `Union[float,List[float]]` precedent (quick 260627-dg0) | Already a proven clean pattern for per-member structures `[VERIFIED: truss2d_model.py:10, app.py:218-233]` |
| Floating panel for the case table | New panel system | Existing `.panel-section` + `setupCardFloat()` | Already present in truss2d (script.js:1786) `[VERIFIED]` |
| Backward-compat JSON load | Migration script | Default-on-read (legacy loads → Dead case), mirroring `memberOverrides` backward-compat | Established additive-load pattern (script.js:1640-1645) |

**Key insight:** The hardest part of a combination generator — re-running analysis for every combination — does NOT exist here because the system is linear. Treat combinations as arithmetic over cached per-case results. Everything else (factor tables, generation, envelope) is small, pure, and testable.

### Combination expressions

**STR-ULS — EN 1990 Eq 6.10:**
```
Ed = Σ_j γG,j·Gk,j  "+"  γQ,1·Qk,1  "+"  Σ_i>1 γQ,i·ψ0,i·Qk,i
```
With UK NA values, one variable action Q1 leading: `1.35·ΣG + 1.5·Q1 + Σ 1.5·ψ0,i·Qi`.
Degenerate Dead+Imposed only → single row `1.35·G + 1.5·Q`. `[CITED: EN 1990 §6.4.3.2 Eq 6.10]`

**Worked example matching CONTEXT D-21** ("ULS-2: 1.35G + 1.5W + 1.05·Imposed"): Wind leading, Imposed accompanying → `1.35·G + 1.5·W + 1.5·0.7·Imposed = 1.35G + 1.5W + 1.05·Imposed`. The 1.05 = γQ·ψ₀ = 1.5 × 0.7. ✓ (internally consistent with the locked example.)

**6.10 leading-action iteration** (the generator's core loop): for *n* variable actions, emit *n* ULS combinations, each taking one variable as leading (full γQ) and the rest accompanying (γQ·ψ₀). With Dead, Imposed, Wind → 2 ULS combos (Imposed leading, Wind leading).

**SLS — characteristic, EN 1990 Eq 6.14b:**
```
Ed = Σ_j Gk,j  "+"  Qk,1  "+"  Σ_i>1 ψ0,i·Qk,i        (all γ = 1.0)
```
Drives the δ_max deflection check already shown in the truss results (script.js:1757-1768). `[CITED: EN 1990 §6.5.3(2)a, Eq 6.14b]`

### Concrete dataclasses (solver_core/loads/combinations.py)

```python
@dataclass
class LoadCase:
    id: str
    name: str
    nature: Nature
    loads: list           # [{nodeId, direction, magnitude}] — unfactored characteristic

@dataclass(frozen=True)
class CombinationTerm:
    case_id: str
    factor: float         # the ONLY place factors live (D-01)

@dataclass
class Combination:
    name: str             # "ULS-2"
    cls: str              # "STR" | "SLS"  (D-17 'class')
    terms: list           # [CombinationTerm]
    expression: str = ""  # human-readable "1.35G + 1.5W + 1.05·Imposed" (D-16/D-21)
    leading: str = ""     # "Wind" — plain leading-action tag (D-16)

@dataclass
class EnvelopeMember:
    member: int                 # 1-based
    max_tension: float          # N
    gov_tension: str            # combination name (D-18)
    max_compression: float      # N (wired now; meaningful after uplift lands)
    gov_compression: str

@dataclass
class EnvelopeResult:
    members: list               # [EnvelopeMember] — member->combo forward map (D-19)
    combo_to_members: dict      # name -> [member ids] reverse index (D-19)
```

### How NEd is computed / which combo is recorded governing

- Per-case `member_forces` come straight from `AnalysisResult.member_forces` (tension +).
- `N_combo[m] = Σ_term factor · per_case_forces[term.case_id][m]`.
- **Governing for max tension:** `argmax` over combinations at member m → record that combination's name.
- **Governing for max compression:** `argmin` (most negative). Recorded now, surfaced as the differentiator once wind-uplift produces load reversal (deferred). In this phase, with downward gravity + wind, the **max-tension** governing combo is what feeds the EC3 tension export (D-21).

This is **portable** (D-22): nothing here is truss-specific. frame2d would feed `member_moments`/`member_shears` arrays through the same superpose/envelope (linear too), so the same `loads/` package serves all solvers.

## Common Pitfalls

### Pitfall 1: Breaking the no-regression gate
**What goes wrong:** Touching `Truss2DRequest`, `solve_truss2d`, or the truss solver to "fit in" combinations changes existing test snapshots.
**Why:** The combination layer is tempting to weave into the existing endpoint.
**How to avoid:** Strictly additive — new module, new endpoint, new response. Run the snapshot suite (`tests/snapshots/baseline/`) and `pytest` (71/71 currently) before/after; both must be byte-identical for existing paths.
**Warning signs:** Any diff in `app.py:213-249` or `truss2d.py`.

### Pitfall 2: Sign / tension convention mismatch in superposition
**What goes wrong:** Member forces flip sign because a downward load was entered as positive somewhere.
**Why:** truss2d is tension-positive (truss2d.py:114); UI loads use "negative = down/left" (script.js:253). Superposition must combine the *solver's* tension-positive `member_forces`, not UI magnitudes.
**How to avoid:** Always superpose `AnalysisResult.member_forces` (already tension-positive). Factors are positive (1.35, 1.5, ψ₀). Hand-check one combo against a direct combined-load solve (Test 4).
**Warning signs:** Envelope tension/compression swapped vs a direct solve.

### Pitfall 3: ψ₀ category ambiguity
**What goes wrong:** Using ψ₀=0.7 for a storage structure (should be 1.0) silently under-factors accompanying imposed.
**Why:** A single "Imposed" nature maps to multiple EN categories with different ψ₀.
**How to avoid:** Implement ψ₀=0.7 default (Cat A–D, H), document it, and design the pack so a per-case category property can override later. Surface the value in the preview expression so the engineer sees it.
**Warning signs:** Engineer working on warehouse/storage; preview shows 1.05·Imposed where 1.5·Imposed expected.

### Pitfall 4: Empty / inactive case included in generation
**What goes wrong:** A case with no loads produces a zero member-force vector but still spawns combinations / appears as leading.
**Why:** Generator iterates all variable actions.
**How to avoid:** Skip empty cases when caching (`if not case.loads: continue`) AND when generating (only generate leading-action permutations over non-empty variable cases). Degenerate Dead+Imposed → single 1.35G+1.5Q (D-12).
**Warning signs:** A "Wind leading" combo appears when no wind loads exist.

### Pitfall 5: Default active case not present on quick path
**What goes wrong:** Quick-solve breaks because loads have no `caseId`.
**Why:** D-03 requires an always-present default Dead case.
**How to avoid:** Create the Dead case at init and on legacy load; never let `loadCases` be empty. `solve()` must work identically whether or not the case table was ever opened.
**Warning signs:** `solve()` errors / empty forceVector after a fresh page load.

### Pitfall 6: Stale results cache after editing a combination
**What goes wrong:** Editing a factor doesn't update the envelope because per-case forces were re-fetched unnecessarily, or not re-combined.
**Why:** Superposition separates the solve (cache per-case) from the combine (cheap).
**How to avoid:** Cache `per_case_forces`; on any combination edit, re-run only `superpose`+`envelope` (no network, no re-solve) — exactly the D-09 benefit.
**Warning signs:** Editing factors triggers an API call or spinner.

### Generate 6.10 STR-ULS combinations (Eurocode pack)
```python
# Source: EN 1990 Eq 6.10 iteration; numbers UK NA Table NA.A1.2(B) + Table A1.1
def generate_eurocode(cases, gamma, psi0, families):
    permanent = [c for c in cases if action_of(c)=="permanent" and c.loads]
    variable  = [c for c in cases if action_of(c)=="variable"  and c.loads]
    combos = []
    if "STR" in families:
        if not variable:
            combos.append(make_combo("ULS-1","STR",
                {p.id: gamma["G_sup"] for p in permanent}))
        for k, lead in enumerate(variable, start=1):
            terms = {p.id: gamma["G_sup"] for p in permanent}
            terms[lead.id] = gamma["Q_sup"]                         # 1.5 leading
            for other in variable:
                if other.id != lead.id:
                    terms[other.id] = gamma["Q_sup"]*psi0[other.nature]  # 1.5·ψ0
            combos.append(make_combo(f"ULS-{k}","STR",terms, leading=lead.nature))
    if "SLS" in families:                                           # characteristic, γ=1.0
        for k, lead in enumerate(variable, start=1):
            terms = {p.id: 1.0 for p in permanent}
            terms[lead.id] = 1.0
            for other in variable:
                if other.id != lead.id:
                    terms[other.id] = psi0[other.nature]
            combos.append(make_combo(f"SLS-{k}","SLS",terms, leading=lead.nature))
    return combos
```
