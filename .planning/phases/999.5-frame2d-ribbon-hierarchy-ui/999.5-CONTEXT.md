# Phase 999.5: Frame2D Ribbon Hierarchy UI - Context

**Gathered:** 2026-05-05
**Status:** Ready for planning (BACKLOG — promote with `/gsd-review-backlog` when sequencing)

<domain>
## Phase Boundary

Replace the current rs3 horizontal wrap-row toolbar in `ui/frame2d/` with a Revit-style ribbon hierarchy: a single-line tab strip at the top of the screen, each tab revealing a panel of buttons grouped by function. Establishes the tab → panel → button mental model so the structure (a) scales as v1.3+ features arrive (load combinations, design solver, etc.) and (b) transfers cleanly to the eventual 3D / Revit-hosted UIs (Phase 12-13 Blender add-on, Phase 7-11 pyRevit pushbuttons), since Revit itself uses the same hierarchy.

**Scope of spike:** frame2d only. Truss2d catches up after frame2d UAT settles. CSS + minimal vanilla JS for tab switching; reuse existing button onclick handlers and `setMode()` logic untouched.

**Out of scope (deferred):**
- Truss2d ribbon migration
- Keyboard shortcuts beyond what already exists
- Contextual tabs (Revit's selection-aware tabs)
- Large/small button size variants
- Persisted last-used tab
- ESC/right-click cancel for drawing modes

</domain>

<decisions>
## Implementation Decisions

### Tab Structure (locked from initial vision)
- **D-00:** 11 tabs in single line at top, in this order:
  1. FILE — Save JSON, Load JSON
  2. EDIT — Edit Node, Undo, Delete, Reset All
  3. GEOMETRY — Add Node, Add Member
  4. SUPPORTS — Fixed, Pin, Roller↔, Roller↕, Spring (5 buttons → 2×3 grid, one slot empty)
  5. NODE LOADS — Force X, Force Y, Moment
  6. MEMBER LOADS — UDL
  7. MEMBER PROPERTIES — Beam/Bar toggle, Pin Left, Pin Right, Per-Member E/I/A
  8. MATERIAL PROPERTIES — Default E, I, A inputs (promoted from inline form per D-02)
  9. SECTION CALCULATOR — Section type + dimensions + Calculate (promoted per D-02)
  10. DISPLAY — Visibility toggles + dialog launcher for advanced settings (per D-01)
  11. SOLVE — Reset View, ▶ SOLVE button

  Future room (later phases): LOAD COMBINATIONS, DESIGN, etc.

### Form-Style Panels
- **D-01:** DISPLAY panel = **hybrid layout**. Inline as compact checkboxes: chkSupports, chkLoads, chkDeflected, chkBMD, chkSFD, chkDiagLabels, chkNodeLabels (touched constantly during modelling). Behind dialog-launcher arrow → "Display Settings" popup: deflection scale, BMD/SFD scale, symbol size, label size. Theme toggle moves out (see D-03).
- **D-02:** **Promote MATERIAL PROPERTIES + SECTION CALCULATOR to first-class tabs in this spike** (not deferred). Reason: vision already lists them as future tabs; promoting now means UAT exercises the full final tab strip and validates real density from day one. 11 tabs total still fits one line.
- **D-03:** Theme toggle moves to **top-right corner of ribbon, outside tab structure** (Revit Quick Access Toolbar pattern). Always visible, no tab switch needed to flip Light/Dark — important since light-mode is being used on the work laptop.

### Panel Layout
- **D-04:** **Strict 2-row, panel grows wide.** Hard rule: every panel is exactly 2 rows tall. As more buttons are added, the panel grows wider indefinitely. If active panel exceeds viewport width, ribbon scrolls horizontally within the panel area only — tab strip stays fixed.
- **D-05:** **Uniform button sizing** (no Revit-style large/small mix). Reuse existing `.tool-btn` styling. Defer large/primary variants to follow-up phase once real-use signal identifies which buttons deserve emphasis.
- **D-06:** Tab strip itself **scrolls horizontally on overflow** (← → arrows at edges). Active tab auto-scrolls into view. Acceptable since work-laptop second-screen aspect is wide; rarely triggers.

### Floating Toolbar Fate
- **D-07:** **Full replace.** Delete the rs3 `<aside class="panel">` structure entirely. Ribbon becomes the only toolbar — no opt-in fallback toggle. Forces clean UAT signal; if ribbon fails we revert via git (one phase = one revertable unit). Aligns with how rs3 itself replaced the prior left-rail layout without preserving fallback.

### Spike Scope
- **D-08:** **Visual + tab-switching only** for first commit set. Includes:
  - CSS for tab strip (single line, scroll on overflow)
  - CSS for panels (strict 2-row grid, uniform buttons)
  - CSS for theme toggle in top-right
  - Minimal vanilla JS for tab click → show/hide panels
  - Form panel layouts for DISPLAY hybrid, MATERIAL PROPERTIES, SECTION CALCULATOR
  - **Reuse all existing button onclick handlers and `setMode()` logic untouched** — no behaviour changes
  Does NOT include: dialog launcher popup behaviour for DISPLAY (panel layout shows where the launcher arrow goes, but click-to-open popup is a follow-up if UAT signals need); active-mode rework; keyboard shortcuts.

### Active Mode Behaviour
- **D-09:** **Keep current `setMode()` click-toggle pattern** with `data-mode` + `.active` class. Clicking another button switches mode. No new ESC/cancel handler in this spike — defer that question to UAT signal.
- **D-10:** **Cross-tab active highlight stays visible.** When a drawing mode is active and the user switches to a different tab, the originating tab gets a subtle "mode active" indicator (small dot or accent border) and the active button retains its `.active` class on the dormant panel. Click on canvas still places nodes/members per active mode. Matches Revit's "active tool persists across tab switches" UX.
- **D-11:** **Default tab on page load = GEOMETRY.** Most-frequent first-action is placing nodes/members. Departs from Revit's FILE-default since FILE here is just Save/Load JSON, rarely a session opener. Persisted last-used tab deferred to follow-up.

### Claude's Discretion
- Exact pixel values for panel `min-width`, tab `padding`, ribbon `height` (use existing design tokens; match rs3 spike density)
- Choice of dialog-launcher arrow glyph (▼ / ↘ / ⤡ — use whichever matches the design token style)
- Internal naming for new CSS classes (`.ribbon`, `.ribbon-tab`, `.ribbon-panel`, etc.)
- HTML structure: ribbon as `<nav class="ribbon">` with `<div class="ribbon-tabs">` + `<div class="ribbon-panels">` or alternative — planner's call
- Whether the tab strip uses `<button>` elements with role="tab" + aria-controls or `<a>` elements (planner picks based on a11y best practice)
- "Mode active" cross-tab indicator visual: dot vs accent border vs glow (planner picks within design token palette)
- Where exactly the dialog-launcher arrow renders within DISPLAY panel (corner, header — planner's call within the hybrid pattern)

</decisions>

## Existing Code Insights

### Reusable Assets
- `.tool-btn` styling (lines 242-260 of style.css) — keep unchanged; ribbon panels render existing buttons with same class
- Design token system in `:root` — has full theme support; ribbon CSS uses tokens only, no new colour literals
- `setMode()` and `data-mode` logic in script.js — preserved untouched per D-08
- `<details>` collapsible card pattern (nwi spike) — superseded by ribbon tabs, can be removed when rs3 panel structure is deleted (D-07)

### Established Patterns
- Mode entry auto-enables matching visibility layer (CLAUDE.md): support modes tick `chkSupports`, load modes tick `chkLoads`, then `draw()` is called. Ribbon must not break this — `setMode()` still owns the toggle logic.
- Light/dark theme toggle persists via localStorage (existing pattern). Theme toggle position changes (D-03) but mechanism stays identical.
- Floating panels (UDL at line 250, Spring at line 271) are positioned `position: fixed` and not part of the ribbon — they overlay the canvas when triggered. No change needed.

### Integration Points
- Ribbon HTML replaces lines 22-189 of `index.html` (the `<aside class="panel">` block). Lines 192-199 (canvas-area) and below stay intact.
- `script.js` adds: tab switching logic (~30 lines), default-tab-on-load (~5 lines), cross-tab active indicator updater (~10 lines). No changes to existing functions.
- The `DISPLAY` dialog-launcher button hooks a new no-op handler in spike (per D-08, popup behaviour is follow-up); planner decides whether to wire a stub `console.log` or leave it as a visible-but-non-functional button with explanatory tooltip.

## Deferred Ideas

These came up during analysis but belong outside this phase — captured so they're not lost.

- **Truss2d ribbon migration** — apply the same ribbon to `ui/truss2d/` after frame2d UAT settles. Likely a sibling phase (999.6 or quick-task) after 999.5 ships.
- **Dialog launcher popup behaviour** — D-08 spike includes the layout (where the launcher arrow appears) but not the click-to-open popup. Promote in a follow-up if UAT shows users clicking the arrow expecting a popup.
- **Active-mode rework** — Revit-style "stays lit until ESC/right-click cancel" (D-09 alternative). Wait for UAT to show whether current click-toggle is sufficient.
- **Contextual tabs** — Revit's selection-aware tabs (e.g., "Modify Member" tab appears when a member is selected). Useful but adds significant state complexity; defer until ribbon basics are validated.
- **Large/small button size variants** — Revit pattern of large primary + small secondary buttons. D-05 went uniform-only for spike; revisit when real-use signal identifies which actions deserve emphasis.
- **Persisted last-used tab** — D-11 defaults to GEOMETRY every reload; localStorage-backed last-used is a small UX win, deferred to follow-up.
- **Keyboard shortcuts** — current UI has none; ribbon could expose Alt-key access to tabs (Revit pattern). Defer until ribbon is validated.
- **Truss2d collapsible-card foot-gun fix** — separate todo at `.planning/todos/pending/2026-05-04-frame2d-toolbar-wrap-improvement.md`; truss2d still has the legacy left-rail layout from before nwi/rs3.
