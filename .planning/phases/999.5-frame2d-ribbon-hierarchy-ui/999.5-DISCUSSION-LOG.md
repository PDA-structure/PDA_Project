## Form-style panels

### Q1: DISPLAY panel rendering

| Option | Description | Selected |
|--------|-------------|----------|
| Hybrid (Recommended) | Critical visibility toggles inline as compact checkboxes; number inputs (scales) + theme toggle behind a dialog-launcher arrow that opens a 'Display Settings' popup. Matches Revit pattern: frequent toggles visible, advanced settings one click away. | ✓ |
| All inline | Render entire form directly in DISPLAY panel — wide, eats horizontal real estate. | |
| All behind launcher | DISPLAY tab shows just a 'Display Settings' button with launcher arrow → popup. Adds friction for frequent toggles. | |

**User's choice:** Hybrid (Recommended)
**Notes:** Decision shaped D-01.

### Q2: MATERIAL PROPERTIES + SECTION CALCULATOR — promote now or defer?

| Option | Description | Selected |
|--------|-------------|----------|
| Promote both now (Recommended) | Add both as first-class tabs in this spike — vision already lists them; building now means full final tab strip is on screen during UAT. 11 tabs total still fits one line. | ✓ |
| Defer to follow-up | Keep current 9 tabs only; promote both later. Smaller spike but doesn't validate full-density layout. | |
| Promote SECTION CALCULATOR only | Compromise: SECTION CALCULATOR self-contained as a tab, MATERIAL PROPERTIES inline in MEMBER PROPERTIES. | |

**User's choice:** Promote both now (Recommended)
**Notes:** Decision shaped D-02 (11-tab spike).

### Q3: Theme toggle placement

| Option | Description | Selected |
|--------|-------------|----------|
| Top-right of ribbon, outside tabs (Recommended) | Theme toggle becomes a fixed icon in top-right corner of ribbon strip. Always visible — matches Revit Quick Access Toolbar pattern. Important since light-mode contrast matters on work laptop. | ✓ |
| Inside DISPLAY tab | Keep as a button in DISPLAY panel (current behaviour). Requires tab switch each flip. | |
| Inside FILE tab | Treat theme as session preference alongside Save/Load JSON. Less intuitive. | |

**User's choice:** Top-right of ribbon, outside tabs (Recommended)
**Notes:** Decision shaped D-03.

## Panel overflow

### Q1: SUPPORTS has 5 buttons; future panels may have 8+. Overflow behaviour?

| Option | Description | Selected |
|--------|-------------|----------|
| Strict 2-row, panel grows wide (Recommended) | Hard rule: every panel exactly 2 rows tall. Panel grows wider as buttons added. Ribbon scrolls horizontally within panel area when oversized; tab strip stays fixed. Most Revit-faithful. | ✓ |
| Cap panel width, scroll inside | Each panel has a max width with internal scroll. Cleaner viewport but max-width is arbitrary. | |
| Allow 3-row override for high-count groups | Default 2 rows, opt-in 3-row for outliers. Compromise. | |

**User's choice:** Strict 2-row, panel grows wide (Recommended)
**Notes:** Decision shaped D-04.

### Q2: Button sizing within panel — uniform or large/small mix?

| Option | Description | Selected |
|--------|-------------|----------|
| Uniform sized (Recommended) | All buttons same size. Reuse existing `.tool-btn` styling. Spike-friendly: zero new CSS variants needed. | ✓ |
| Large + small mix (Revit-style) | Primary action large, secondary small. More Revit-faithful but design overhead during spike. | |

**User's choice:** Uniform sized (Recommended)
**Notes:** Decision shaped D-05.

### Q3: Tab strip overflow on small screens (11+ tabs may not fit)?

| Option | Description | Selected |
|--------|-------------|----------|
| Horizontal scroll on tab strip (Recommended) | Tab strip scrolls horizontally with ← → buttons at edges; active tab auto-scrolls into view. Won't trigger often on wide work-laptop second screen. | ✓ |
| Compact text + tooltips | Reduce font and abbreviate labels; tooltip shows full name. Trades legibility for visibility. | |
| Defer — worry about it on UAT | Ship without overflow handling; fix in follow-up if it becomes a problem. | |

**User's choice:** Horizontal scroll on tab strip (Recommended)
**Notes:** Decision shaped D-06.

## Active mode + default tab

### Q1: Active drawing mode behaviour

| Option | Description | Selected |
|--------|-------------|----------|
| Keep current click-toggle (Recommended) | Same setMode() logic, same data-mode + .active class highlighting. Defer ESC/cancel question to UAT signal. | ✓ |
| Add ESC to cancel mode | One-line keydown handler clears mode. Low-risk Revit/CAD norm. | |
| Full Revit-style stays-lit-until-cancel | Mode stays active forever until ESC/right-click. High UX departure; high risk in spike. | |

**User's choice:** Keep current click-toggle (Recommended)
**Notes:** Decision shaped D-09.

### Q2: Cross-tab active highlight behaviour

| Option | Description | Selected |
|--------|-------------|----------|
| Highlight stays on dormant tab (Recommended) | Originating tab gets a subtle 'mode active' indicator (small dot or accent border); active button retains .active class on dormant panel. Click on canvas still places nodes per active mode. Matches Revit's "active tool persists across tab switches" UX. | ✓ |
| Auto-return to mode's tab | Switching back to canvas auto-pops the mode's tab. Aggressive, may surprise users. | |
| Mode clears on tab switch | Switching tabs cancels mode. Loses workflow continuity. | |

**User's choice:** Highlight stays on dormant tab (Recommended)
**Notes:** Decision shaped D-10.

### Q3: Default tab on page load

| Option | Description | Selected |
|--------|-------------|----------|
| GEOMETRY (Recommended) | Most users start by placing nodes/members. Add Node + Add Member one click away on first load. Matches actual workflow. | ✓ |
| FILE | Familiar Revit pattern but our FILE tab only has Save/Load JSON, rarely first action. | |
| Last-used tab (persisted) | Remember last active tab via localStorage. Small UX win; deferred to follow-up. | |

**User's choice:** GEOMETRY (Recommended)
**Notes:** Decision shaped D-11. If UAT signals this is wrong it's a one-line change.
